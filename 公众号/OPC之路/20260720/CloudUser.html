用 CloudKit，零成本拥有自己的用户体系

做独立开发，最容易被低估的一件事，就是“用户体系”。

很多人以为，用户体系只是做一个登录页面：输入手机号，收验证码，然后进入 App。

但真正做起来之后才会发现，登录页面只是开始。后面还有用户 ID、账号找回、多设备同步、数据归属、隐私保护、服务器、数据库、短信费用，以及一大堆需要长期维护的后台逻辑。

对于一个独立开发者来说，这些事情很容易让一个原本简单的产品，变成一个需要持续运维的系统。

有没有一种方式，不搭建服务器，不购买短信服务，也不自己维护一套账号密码系统，就能让 App 拥有基本的用户能力？

有。

答案就是 Apple 提供的 CloudKit。

一、CloudKit 是什么

CloudKit 是 Apple 提供的云端数据服务。它和 iCloud 账户、Apple 设备体系深度结合，可以帮助我们保存和同步 App 数据。

它提供了几类基础能力：

  • 用户身份识别
  • 公共数据库
  • 私有数据库
  • 文件和图片存储
  • 数据查询与修改
  • 数据变化订阅
  • 多设备数据同步

最吸引独立开发者的地方是：我们不需要先购买服务器、配置数据库、设计密码登录接口，也不需要自己处理短信验证码。

用户已经拥有 Apple 账户，而 CloudKit 可以在用户登录 iCloud 的前提下，为 App 提供一个稳定的云端身份。

换句话说，我们可以直接利用 Apple 已经建设好的基础设施，给自己的 App 建立一套用户和数据体系。

二、CloudKit 能不能当作用户系统

这里需要先说清楚一个概念：CloudKit 不是传统意义上的“账号密码系统”。

它不会替我们提供一个邮箱密码登录页面，也不会让我们拿到用户的 Apple 账户密码。它做的事情是：在用户允许的范围内，告诉 App 当前设备对应的 CloudKit 用户身份。

整个关系大概是这样：

Apple 账户
    ↓
iCloud 身份
    ↓
CloudKit 用户 Record ID
    ↓
App 自己的 UserProfile
    ↓
用户的业务数据

在 App 中，我们通常不需要保存用户的邮箱或真实姓名，而是使用 CloudKit 返回的用户 Record ID 作为当前用户在这个 Container 中的身份标识。

然后，再在自己的数据库中建立一个 UserProfile，保存昵称、头像、设置、会员状态等业务信息。

这样一来，CloudKit 负责“这个人是谁”，App 负责“这个人的资料和业务数据是什么”。

三、第一次启动时发生了什么

一个最小可用的用户初始化流程，可以分成几步：

  1. App 启动后检查 iCloud 账户状态。
  2. 获取当前 CloudKit 用户的 Record ID。
  3. 查询当前用户是否已经有 UserProfile。
  4. 如果没有，就创建一条用户资料。
  5. 加载这个用户的收藏、设置、历史记录等业务数据。
  6. 根据需要建立 CloudKit Subscription，监听数据变化。

在 Swift 中，获取当前用户身份的代码并不复杂：

import CloudKit

let container = CKContainer(identifier: "iCloud.com.example.myapp")

container.accountStatus { status, error in
    guard status == .available else {
        // 用户没有登录 iCloud，或者当前账户暂时不可用
        return
    }

    container.fetchUserRecordID { recordID, error in
        guard let recordID else {
            return
        }

        print("当前用户：\(recordID.recordName)")
    }
}

实际项目中，还需要处理网络失败、iCloud 未登录、账户切换和 CloudKit 服务暂时不可用等情况。不要把“成功拿到用户 ID”当作永远成立的前提。

四、CloudKit 的三种数据库

CloudKit 中最重要的设计之一，是选择正确的数据库。

1. Private Database：用户自己的数据

Private Database 适合保存用户个人数据，例如：

  • App 设置
  • 收藏记录
  • 阅读历史
  • 私人笔记
  • 用户创建但不想公开的内容

每个用户都有自己的 Private Database。数据天然和当前用户关联，其他用户无法直接读取。

这对于工具类 App 非常方便。我们不需要在每一条数据上重复设计复杂的用户权限，只需要把个人数据放进用户自己的数据库即可。

2. Public Database：所有用户共享的数据

Public Database 适合保存公共内容，例如：

  • 文章
  • 公共配置
  • 分类信息
  • 排行榜
  • 用户公开发布的内容

公共数据库中的数据需要认真设计权限。哪些数据所有人可以读取，哪些数据只有创建者可以修改，都应该在数据模型和 CloudKit Dashboard 中明确配置。

3. Shared Database：共享和协作数据

Shared Database 适合家庭、团队或多人协作场景，比如共享清单、共同编辑的内容等。

如果 App 只是保存用户个人数据，通常先使用 Private Database 就够了；如果是公共内容，再增加 Public Database。不要一开始就把所有数据都放到公共数据库中。

一个简单的判断方法是：

私人的数据放 Private Database，公开的数据放 Public Database，需要协作的数据再考虑 Shared Database。

五、如何设计自己的用户资料

CloudKit 会提供用户身份，但不会替我们定义 App 用户资料。

我们可以在自己的模型中设计一个 UserProfile，例如：

UserProfile
├── recordID
├── nickname
├── avatar
├── createdAt
├── updatedAt
├── isPro
└── settings

然后，用户的其他业务数据通过用户身份建立关联：

UserProfile
├── 收藏记录
├── 阅读历史
├── App 设置
├── 用户笔记
└── 订阅状态

需要注意的是，不要把邮箱、真实姓名等敏感信息当作业务主键。Record ID 更适合做内部关联，而昵称、头像等展示资料应该由用户自己设置或授权提供。

另外，Record ID 的使用范围也要理解清楚：它适用于当前 CloudKit Container，不应该简单地当成所有 App、所有平台通用的用户 ID。

六、CloudKit 如何实现多设备同步

当用户在 iPhone 和 iPad 上使用同一个 Apple 账户，并且 App 使用同一个 CloudKit Container 时，保存到 Private Database 的数据就可以在设备之间同步。

例如，用户在 iPhone 上收藏了一篇文章：

iPhone 修改数据
    ↓
保存到 CloudKit Private Database
    ↓
CloudKit 记录发生变化
    ↓
iPad 收到变化通知
    ↓
iPad 更新本地数据

真正好用的同步体验，通常不是每次打开页面都从云端重新读取，而是采用“本地缓存 + 云端同步”的方式：

  • 页面优先展示本地数据
  • 有网络时上传本地修改
  • 云端有变化时拉取更新
  • 同步失败时保留待同步状态
  • 多设备同时修改时执行冲突处理

CloudKit 更适合最终一致的数据同步，而不是实时数据库。对于聊天、实时协作等场景，仍然需要考虑更专业的后端方案。

七、CloudKit 的成本：真的完全免费吗

标题里说“零成本”，主要是站在独立开发者的角度：不需要额外购买一台服务器，也不需要从零开始维护后端用户系统。

但它并不代表完全没有任何成本。

可能节省的成本包括：

  • 服务器租赁费用
  • 数据库维护成本
  • 短信验证码费用
  • 邮箱登录服务费用
  • 用户密码和账号找回系统
  • 后端基础设施的运维时间

仍然需要考虑的成本包括：

  • Apple Developer Program 账号费用
  • CloudKit 的存储、请求和流量配额
  • 大规模用户增长后的容量和费用
  • 对 iCloud 账户的依赖
  • CloudKit 数据模型和权限配置的学习成本

所以，更准确的说法是：

CloudKit 是低成本、免运维的用户和数据基础设施，而不是绝对意义上的永久免费后端。

对于早期产品、个人工具和小规模 App 来说，这种成本结构已经非常友好。

八、CloudKit 适合哪些 App

CloudKit 特别适合以下类型的产品：

  • 纯 Apple 生态 App
  • 笔记、阅读、收藏类 App
  • 效率工具和个人管理工具
  • 需要多设备同步的 App
  • 独立开发者的 MVP 产品
  • 不需要复杂服务端计算的应用

它不太适合以下场景：

  • 必须同时支持 Android、Web 和小程序的产品
  • 以手机号为核心身份的业务
  • 即时通讯和强实时协作
  • 复杂电商和订单系统
  • 大型社区和高频公共内容平台
  • 依赖复杂后端任务、风控和数据分析的产品

如果未来确实需要支持多个平台，也可以采用混合方案：CloudKit 负责 Apple 平台上的个人数据同步，自建后端负责跨平台用户、订单和复杂业务。

九、CloudKit 的真正价值

CloudKit 最大的价值，并不是“免费”两个字。

它真正解决的是：一个人，也可以拥有过去需要后端团队才能维护的基础能力。

你不需要先研究服务器部署，不需要先购买短信服务，也不需要先写一套账号密码和找回流程。只要你的产品主要服务于 Apple 用户，就可以利用用户已有的 Apple 账户，快速建立身份和数据同步能力。

当然，CloudKit 不是万能的。

它有平台限制，有数据配额，也不适合所有复杂业务。但对于一个独立开发者来说，它非常适合作为第一套后端基础设施：足够简单，足够便宜，也足够支撑很多真实产品。

过去，做一个带用户系统的 App，往往意味着先搭建后端。

现在，很多时候可以反过来：

先把产品做出来，先让用户用起来，等真正遇到 CloudKit 解决不了的问题，再引入更复杂的服务。

这可能就是今天独立开发最重要的变化之一。

一个人，也可以用很低的成本，做出完整的产品。
